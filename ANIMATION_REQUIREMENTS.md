# 网站动画规范（后续实现需遵循）

## 总体原则
- 动画要“有信息含量”，不能只是卡片高亮或简单移动。
- 每个知识点必须有对应的动画展示，且能解释核心机制。
- 动画与文字说明要一致，突出“原理 + 过程 + 关键结论”。
- 保持视觉风格统一：暖色系、圆角、轻阴影、节奏舒缓。

## 交互与结构
- 优先采用 **分步骤** 动画（时间线/步骤卡片），支持播放/暂停/上一步/下一步。
- 明确路径或流程时，优先使用 **泳道图/流程图/时序图**。
- 切换步骤时必须有清晰的高亮/焦点提示。

## 表达形式要求
- 使用 **示例驱动**：通过具体 key/参数/数值帮助理解（例如缓存淘汰策略）。
- 有必要时展示 **LeetCode 模式的 C++ 实现**（如 LRU/LFU），并支持代码高亮。
- 动画内容要体现“机制差异”（例如 HTTP 1.1 队头阻塞 vs HTTP/2 多路复用）。

## 视觉与布局
- 避免依赖 `position: absolute` 导致高度无法撑开，防止与下方模块重叠。
- 小屏仍需可读性：布局缩放但不失真，必要时压缩气泡/间距。
- 不使用 “hover-only” 作为关键动画触发方式，移动端需可操作。

## 质量标准
- “看起来更丰富、更有意思”是底线：至少包含
  - 多元素协同动画（非单一元素）
  - 具象流程展示（节点/流向/队列/状态）
  - 清晰对比（不同策略/版本/阶段的差异）
- 禁止简单示意图替代关键原理动画。

## 参考实现示例
- TCP 三次握手/四次挥手：泳道时序图 + 行式消息气泡
- HTTP 版本演进：1.1 队头阻塞、2 多路复用帧、3 QUIC + 0-RTT
- Redis 淘汰策略：样本 key 池 + TTL/LRU/LFU 选择对比 + C++ 代码
